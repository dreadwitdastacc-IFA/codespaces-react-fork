📚 CSS Cascade & Equal Specificity

When Specificity is Equal, the Cascade Decides

Example 1: Same Specificity (0-0-1)

h1 0-0-1 1st
h1 0-0-1 2nd ✓ WINS
Specificity is identical: Both are 0-0-1

Resolver: Source order (cascade) determines the winner

Result: The rule appearing later in the stylesheet wins

Example 2: Element Chain Specificity

body h1 0-0-2 1st
html h1 0-0-2 2nd ✓ WINS
Both have: 2 element selectors = 0-0-2

Specificity is a complete tie

Winner: The rule defined later in the source code

Depth doesn't matter: body → h1 is same specificity as html → h1

Live Preview

I'm orange because html h1 comes later (0-0-2)

✓ html h1 (later) beats body h1 (earlier) due to cascade

Example 3: :where() vs :is()

:where(#test) h2 0-0-1 1st
:is(#test) h2 1-0-1 2nd ✓ WINS
Key Differences:
• :where() = zero specificity (0-0-1)
• :is() = takes highest argument (1-0-0) + element (1-0-1)

Winner: :is() always wins due to higher specificity regardless of order

Live Preview

I'm pink because :is(#test) wins (1-0-1 > 0-0-1)

✓ Specificity determines winner (not cascade order here)

Example 4: @layer Cascade

@layer base 0-0-1 Declared 1st
@layer theme 0-0-1 Declared 2nd ✓
@layer declaration order:
@layer base, theme; (declares order)

Rules: Later layers override earlier ones, regardless of source order within the layer

Winner: theme layer wins even if defined before base

Live Preview

I'm cyan (theme layer wins) even though base was defined first

✓ Layer order (theme after base) determines winner, not source order

Example 5: !important Declaration

p.example5-normal 0-1-1 Normal
p.example5-important 0-1-1 !important ✓ WINS
⚠️ Important Rule: !important creates a separate cascade tier
It trumps all normal declarations regardless of specificity
Use sparingly! It breaks normal cascade

I'm green because !important overrides everything

⚠️ !important should be avoided - use @layer to manage cascade instead

🔄 CSS Cascade Order (Highest to Lowest Priority)

  1. Transition declarations (animations)
  2. User-origin !important (user styles with !important)
  3. Author-origin !important (stylesheet !important)
  4. Later @layers ← Most important layer concept!
  5. Earlier @layers
  6. Inline styles (style="...")
  7. ID selectors (1-0-0)
  8. Class/Attribute/Pseudo-class (0-1-0)
  9. Element selectors (0-0-1)
  10. Inherited values
  11. Browser defaults

✨ Key Cascade Principles

  • Specificity is compared left-to-right: One ID beats any number of classes
  • Source order matters only when specificity is equal
  • Later rules win when specificity and layer are equal
  • @layer declaration order determines cascade - not physical position in file
  • !important reverses cascade order (avoid when possible)
  • Inline styles beat selectors but lose to !important
  • Inherited values have lowest priority (can always override)
  • Use @layer instead of !important for better maintainability

💡 Best Practices

  • Prefer lower specificity: Makes styles easier to override
  • Avoid ID selectors in stylesheets: Reserve them for JavaScript
  • Use @layer for organization: Reset/Base → Theme → Components → Utilities
  • Keep cascade in mind: The last rule can override earlier ones
  • Document your cascade layers: Makes team collaboration easier
  • Test cascade behavior: Different media queries or feature flags
